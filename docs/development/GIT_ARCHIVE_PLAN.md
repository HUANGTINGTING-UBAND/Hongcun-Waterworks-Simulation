## Git LFS recommendation

Git LFS is recommended for `*.blend` files because Blender binaries are non-diffable and each revision otherwise remains in normal Git history. The current recommended Blender binaries total only about 2.09 MiB, so LFS is not required by individual file-size limits; the recommendation is based on expected binary-history growth.

Current environment:

- Git LFS command: not installed
- `.gitattributes`: not present
- Existing LFS tracking rules: none

Do not stage the Blender files until the repository owner decides whether to use LFS. If approved, the future setup sequence is:

```bash
brew install git-lfs
git lfs install
git lfs track "*.blend"
```

The generated `.gitattributes` must be reviewed and committed with the Blender files. If LFS is not adopted, the current approximately 2.63 MiB commit remains technically reasonable, but repeated Blender revisions will enlarge repository history.

## Pre-commit checks

Before committing:

1. Run `git status --short` and confirm no terrain, GIS, nature, `.blend1`, or temporary render file is staged.
2. Run `git diff --cached --name-status` and compare it with the explicit list above.
3. If LFS is enabled, run `git lfs ls-files` and confirm every staged `.blend` is listed.
4. Recalculate SHA-256 for the approved prefab and compare it with `architecture_asset_manifest.json`.
5. Verify that `preview_render.png` remains tracked normally and is not matched by temporary-render rules.

Suggested commit message:

```text
chore: archive frozen Hui architecture assets
```
