# 宏村水脉

# Hongcun Water Pulse

# GDD09_Codex_Development_Workflow.md

版本： Project Bible v2.1

文档类型： Game Design Document

用途：

定义《宏村水脉》项目中 Codex、AI Agent 与开发人员的协作流程。

目标：

建立从设计文档到代码实现的稳定开发闭环。

------------------------------------------------------------------------

# 1. Codex角色定义

Codex在本项目中的身份：

项目执行工程师。

负责：

-   阅读项目文档；
-   分析开发任务；
-   编写代码；
-   执行测试；
-   更新文档。

Codex不是：

-   历史内容创造者；
-   自由策划者；
-   未授权架构修改者。

------------------------------------------------------------------------

# 2. AI开发闭环

标准流程：

    需求

    ↓

    GDD设计

    ↓

    CODEX_TASK

    ↓

    代码实现

    ↓

    测试验证

    ↓

    文档更新

    ↓

    Git Commit

任何功能必须经过完整流程。

------------------------------------------------------------------------

# 3. 任务拆分规范

每个开发任务必须包含：

    TASK ID

    目标

    背景

    输入文件

    修改范围

    输出结果

    验收标准

    测试方法

示例：

    TASK-WATER-001

    目标：

    实现水节点系统。

    输入：

    GDD04

    data/water/

    输出：

    WaterNode.ts

    验收：

    能够加载水节点数据。

------------------------------------------------------------------------

# 4. GitHub Issue规范

Issue格式：

    [Feature]

    [System]

    [Task ID]

    Description

    Acceptance Criteria

示例：

    [Feature]

    Water Simulation

    [Task]

    TASK-WATER-002

    实现高程驱动水流。

------------------------------------------------------------------------

# 5. 分支规范

推荐：

    main

    develop

    feature/*

    fix/*

    docs/*

规则：

-   main保持稳定版本；
-   develop用于开发集成；
-   feature用于单功能开发。

------------------------------------------------------------------------

# 6. Commit规范

格式：

    type(scope): description

示例：

    feat(water): add canal flow engine

    fix(map): correct terrain loader

    docs(gdd): update NPC design

------------------------------------------------------------------------

# 7. AI修改原则

禁止：

-   一次修改大量无关模块；
-   删除已有系统；
-   改变核心架构；
-   未审核新增历史内容。

必须：

-   小任务；
-   小修改；
-   可测试；
-   可回滚。

------------------------------------------------------------------------

# 8. 文档同步规则

代码变化后必须检查：

    README.md

    PROJECT_CONTEXT.md

    AGENTS.md

    GDD

    DATA_SCHEMA

如果系统逻辑改变：

必须同步更新文档。

------------------------------------------------------------------------

# 9. 测试规范

每个核心系统需要：

## 单元测试

验证：

-   算法；
-   数据处理。

## 集成测试

验证：

-   系统连接。

## 视觉测试

验证：

-   三维表现；
-   用户交互。

------------------------------------------------------------------------

# 10. AI协作安全规则

禁止：

-   删除历史数据库；
-   修改历史真实性规则；
-   替换技术架构；
-   添加未经确认的人物和事件。

------------------------------------------------------------------------

# 11. 开发优先级

按照宏村核心逻辑：

    1 地形加载

    ↓

    2 水系统

    ↓

    3 时间系统

    ↓

    4 建筑系统

    ↓

    5 NPC系统

    ↓

    6 任务系统

    ↓

    7 社会经济系统

    ↓

    8 美术优化

原因：

水是宏村聚落形成的基础。

# 13. 项目开发阶段

## Phase 0

项目基础

完成： - 工程框架； - 数据结构； - 文档体系。

## Phase 1

数字宏村世界

完成： - 三维地形； - 自然水系； - 摄像机系统。

## Phase 2

水利模拟

完成： - 水流； - 水圳； - 月沼； - 灾害。

## Phase 3

历史互动

完成： - NPC； - 任务； - 工程建设。

## Phase 4

完整体验

完成： - 聚落演化； - 南湖阶段； - 四季系统； - 摄影模式。

------------------------------------------------------------------------

# 14. 验收标准

完成后：

团队能够：

-   使用AI持续开发；
-   保持历史一致；
-   避免架构混乱；
-   快速扩展功能。

------------------------------------------------------------------------

# 15. 设计总结

Codex不是替代设计者。

Codex是帮助项目稳定落地的工程伙伴。

所有开发行为必须服务于：

建立一个真实、可交互、可持续扩展的宏村数字历史系统。

END
